# 朋友圈文案生成器 - MVP 产品需求文档

**项目名称**：朋友圈文案生成器  
**版本**：v1.0 (MVP)  
**日期**：2026-03-31  
**目标平台**：微信小程序

---

## 1. 项目背景

### 1.1 痛点分析

每次发朋友圈前，用户需要经历三个最耗时的环节：
1. **选图**：从几十张照片中挑选9张最合适的
2. **排版**：尝试各种排列组合，找到视觉最和谐的9宫格
3. **写文案**：绞尽脑汁想一句"不土但有辨识度"的文案

### 1.2 产品愿景

让用户只负责把修好的图片上传，系统自动完成：
- 智能排序9宫格
- 生成多种风格的文案草稿
- 一键复制到朋友圈

---

## 2. 产品目标

### 2.1 MVP 核心目标

| 目标 | 具体描述 |
|------|----------|
| **完成度** | 打通从上传图片到生成文案的完整链路 |
| **可用性** | 用户能在30秒内完成一次完整使用 |
| **质量** | 文案生成结果"不土"，有基本的场景适配能力 |

### 2.2 成功指标

- 用户一次完整操作流程 < 30秒
- 文案生成满意度（用户自评）≥ 3.5/5
- 9宫格排序视觉和谐度 ≥ 70%

---

## 3. 用户画像

| 类型 | 特征 | 核心需求 |
|------|------|----------|
| **摄影爱好者** | 经常修图、拍图，发朋友圈频率高 | 快速把修好的图发出去，不想在文案上花时间 |
| **精致生活者** | 注重朋友圈质感，对文案和排版有要求 | 希望文案"有格调"，排版"有品味" |
| **职场人士** | 偶尔发工作/生活相关内容 | 希望文案得体、不突兀 |

## 5. 核心功能详情

### 5.1 图片上传模块

**用户故事**：作为用户，我希望快速上传修好的图片，这样可以开始生成内容。

**功能描述**：
- 支持从相册选择1-9张图片
- 显示已选图片的缩略图网格
- 支持删除已选图片
- 图片大小限制：单张 ≤ 10MB

**交互流程**：
1. 用户进入首页，点击"选择图片"
2. 打开相册选择器，选择1-9张图片
3. 确认后进入下一步

---

### 5.2 9宫格排序模块

**用户故事**：作为用户，我希望系统自动把图片排得好看，这样我不用自己试各种组合。

**排序策略（MVP）**：
1. **主图优先**：识别最突出的一张放在中间（第5格）
2. **色调过渡**：根据图片主色调进行渐变排列
3. **内容平衡**：避免相似内容的图片集中在一起

**交互**：
- 显示排序后的9宫格预览
- 支持拖拽调整位置（可选）
- 点击"重新排序"可再次生成

---

### 5.3 文案生成模块

**用户故事**：作为用户，我希望系统根据图片生成几种文案，这样我可以选一个最喜欢的。

**场景分类（MVP）**：
| 场景 | 说明 |
|------|------|
| 旅行 | 风景、地标、旅途 |
| 生活 | 美食、日常、聚会 |
| 工作 | 办公、会议、成就 |
| 其他 | 无法明确分类的内容 |

**文案风格（MVP）**：
| 风格 | 特点 | 示例 |
|------|------|------|
| 简洁 | 短平快，不啰嗦 | "今天天气真好。" |
| 文艺 | 有画面感，带点诗意 | "风路过的时候，也停了停。" |
| 幽默 | 轻松有趣，带点自嘲 | "P图两小时，发圈一分钟。" |
| 走心 | 真诚，有温度 | "记录这一刻的小确幸。" |

**交互流程**：
1. 系统自动识别场景
2. 用户可手动调整场景
3. 点击"生成文案"
4. 显示3-5条不同风格的文案
5. 用户选择一条，可手动修改

---

### 5.4 导出模块

**用户故事**：作为用户，我希望一键把文案复制好，这样可以直接去朋友圈粘贴。

**功能**：
- 点击文案卡片即可复制到剪贴板
- 显示"已复制"提示
- 可选：保存排序后的图片到相册

## 7. 技术架构

### 7.1 技术选型

| 层级 | 技术方案 | 说明 |
|------|----------|------|
| **前端** | Taro + React + TypeScript | 跨端框架，支持微信小程序 |
| **UI组件** | Taro UI 或 NutUI | 小程序端UI组件库 |
| **图片处理** | 微信小程序 Canvas API | 前端排序、拼接 |
| **AI服务** | 第三方AI API | 图片理解、文案生成 |
| **存储** | 微信小程序本地缓存 | 用户偏好、历史记录 |

### 7.2 系统架构

```
┌─────────────────────────────────────────┐
│         微信小程序 (Taro)               │
│  ┌─────────────┐  ┌─────────────────┐ │
│  │   页面层     │  │   组件层         │ │
│  │ - 首页      │  │ - 图片选择器    │ │
│  │ - 编辑页    │  │ - 9宫格预览     │ │
│  │             │  │ - 文案卡片       │ │
│  └─────────────┘  └─────────────────┘ │
│  ┌───────────────────────────────────┐  │
│  │           业务逻辑层               │  │
│  │ - 图片排序算法                    │  │
│  │ - 场景识别逻辑                    │  │
│  │ - 文案生成流程                    │  │
│  └───────────────────────────────────┘  │
└─────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────┐
│              后端服务                    │
│  ┌───────────────────────────────────┐  │
│  │         AI 服务集成                │  │
│  │ - 图片理解 (Vision API)           │  │
│  │ - 文案生成 (LLM API)              │  │
│  └───────────────────────────────────┘  │
└─────────────────────────────────────────┘
```

### 7.3 AI 服务方案（MVP）

**备选方案**：

| 方案 | 优点 | 缺点 |
|------|------|------|
| **方案A：直接调用第三方 API** | 开发快，无需维护后端 | 成本较高，依赖第三方 |
| **方案B：自建后端 + AI** | 可控性强，可优化 | 开发周期长，成本高 |

**MVP 选择：方案A**

## 10. 风险与应对

| 风险 | 影响 | 概率 | 应对方案 |
|------|------|------|----------|
| AI API 费用高 | 高 | 中 | MVP 限次数，后续考虑优化 |
| 文案质量不达预期 | 高 | 中 | 多提供几种风格，让用户选 |
| 小程序审核不通过 | 中 | 低 | 提前了解审核规范，避免敏感内容 |
| 图片处理性能问题 | 中 | 中 | 前端做压缩、限制图片大小 |
